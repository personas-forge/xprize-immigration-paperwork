## 1. Orchestrator migration stalled at 4/6 — `draft`, `rfe`, `evidence/categorize` still hand-roll the full pipeline it was built to own
- **Severity**: critical
- **Category**: duplication
- **File**: src/lib/ai/operation.ts:181 (built); src/app/api/draft/route.ts:52-297, src/app/api/rfe/route.ts:36-183, src/app/api/evidence/categorize/route.ts:35-146 (hand-rolled duplicates)
- **Scenario**: A maintainer changes a cross-cutting AI-pipeline invariant — e.g. the 402 body shape, the 429 `Retry-After` contract, or the charge-then-reclaim-to-mock recovery. They edit `executeAiOperation` (the documented owner) and assume every charged AI route inherits it. Three of the five charged routes do not.
- **Root cause**: `executeAiOperation` was built (ADR-0004 #10) as "the foundation the AI routes migrate onto" (CHANGELOG 0.4.0). Only `/api/qualify` (#12) and `/api/guidance` (#13) were migrated — their route comments self-label "task 4/6" and "task 3/6", confirming a 6-task plan whose remaining route migrations (draft, rfe, evidence) were never completed. Those three routes still inline the identical stage sequence the orchestrator centralizes: `request.json()` → 400; rate-limit-before-charge → 429 + `Retry-After` + DISCLAIMER; `chargeForOperation` → 401 / 402 (cost/balance/DISCLAIMER); `getLlm()` → try/parse/guard, `reclaim()` + mock-with-`source:"mock"` on throw OR unusable output; best-effort persist with `saveFailed`. The orchestrator's own docstring (operation.ts:21-29) asserts "The five AI endpoints (`draft`, `rfe`, `qualify`, `guidance`, `evidence/categorize`) all hand-implement the SAME pipeline … (~500 lines duplicated). This module owns that pipeline once" — but in the current tree it only owns it for 2 of 5.
- **Impact**: The single highest-value finding. The abstraction exists and is unit-tested (13 cases) yet ~60% of its intended call sites bypass it, so the duplication it was created to eliminate is still live in ~300 hand-rolled lines across three files. Every invariant must be maintained in four places (orchestrator + 3 routes), and they already drift (see finding 2). This is a half-finished refactor, not a stable architecture.
- **Verification**: `grep executeAiOperation` across `src/app/api` → exactly 2 importing/calling routes: `guidance/route.ts:9,36` and `qualify/route.ts:12,36`. `grep` for the hand-rolled markers (`chargeForOperation`, `checkRateLimit`, `insufficient_tokens`, `DISCLAIMER`) across `src/app/api` → 6 route files; subtracting the 2 adopters leaves `draft`, `rfe`, `evidence/categorize`, `draft/save` still inlining the preamble. Read all six route files end-to-end: draft (lines 104-296), rfe (74-182), categorize (71-145) reproduce the orchestrator's stages 3-7 by hand. Only the AI routes that actually charge+call-a-model are in scope here (draft/save is charge-free; see finding 2).
- **Fix sketch**: Resume the ADR-0004 migration: convert `rfe` and `evidence/categorize` to `AiOperationSpec`s first (they map cleanly — single op key, one prompt, one guard, one persist). `draft` needs a small extension because it bills two ops and has a two-branch (full vs `focus`) flow — `operation` is already a function (operation.ts:107) and `persist` can host the merge logic, but the focus-merge "no stored base draft" surfacing (draft/route.ts:236-245) must be preserved via `onPersistError`. Behavior-preserving only: do NOT alter the charge/reclaim/disclaimer semantics during the move; pin each migrated route with a route-level test mirroring the existing 13 orchestrator cases. Flag as a non-trivial change on money + UPL paths — land one route per PR.

## 2. 429 disclaimer drift: `draft/save` omits the DISCLAIMER the orchestrator (and every other route) guarantees
- **Severity**: high
- **Category**: duplication
- **File**: src/app/api/draft/save/route.ts:73-78
- **Scenario**: A client hits the per-window cap on the draft-save rescue endpoint. It receives `{ error: "rate_limited", retryAfterSec }` with **no** `disclaimer` field — whereas the same 429 from `/api/draft`, `/api/rfe`, `/api/guidance`, `/api/qualify`, `/api/evidence/categorize`, and `executeAiOperation` itself all include `disclaimer: DISCLAIMER`. A UI that reads `body.disclaimer` to stamp the not-legal-advice notice renders nothing on this one path.
- **Root cause**: `draft/save` is the one charged-family route that was never migrated and never even imported `DISCLAIMER` (the import list, lines 1-12, has no `@/features/guidance` / `@/lib/result` disclaimer import). Its hand-rolled 429 (lines 74-77) therefore can't include it. The orchestrator's invariant #1 (operation.ts:36-37, "DISCLAIMER is present on every error body it emits") is exactly the rule this route violates. This confirms the Wave-1 observation that the save route's 429 omits the disclaimer.
- **Impact**: A genuine UPL-safeguard inconsistency on a compliance-sensitive product, caused by the unfinished centralization: the contract the orchestrator enforces uniformly is silently weaker on the one route that bypasses it. Low blast radius (rate-limit path only) but it is a real divergence, not cosmetic.
- **Verification**: Read all six routes. 429 bodies: draft:108, rfe:78, guidance (via orchestrator):224, qualify (via orchestrator):224, categorize:75 all include `disclaimer: DISCLAIMER`; `draft/save:75` does not. `grep DISCLAIMER src/app/api/draft/save/route.ts` → 0 hits. Orchestrator test "rate-limit miss → 429 … + DISCLAIMER" (operation.test.ts:123-145) asserts the field is present, pinning the contract the save route breaks.
- **Fix sketch**: Narrowest fix — import `DISCLAIMER` and add it to the save route's 429 body to match. Note `draft/save` is NOT a clean `executeAiOperation` adopter (it neither charges nor calls a model — it is a persistence-only rescue, route.ts:14-25), so it should not be force-fit onto the orchestrator; either share a tiny `rateLimited429(retryAfterSec)` helper that always stamps the disclaimer, or add the field inline. Pure additive change; no charge/model semantics touched.

## 3. `ChargeOutcome` / `AuthUser` / `OperationLlm` are hand-mirrored copies of real infra types, decoupled only for the test runner
- **Severity**: medium
- **Category**: structure
- **File**: src/lib/ai/operation.ts:53-68, 135-147
- **Scenario**: `ChargeResult` in `@/lib/tokens/guard` gains a field (or a new failure `reason`). The orchestrator's local `ChargeOutcome` mirror is not updated; the `defaultDeps` adapter (operation.ts:160) keeps compiling because `chargeForOperation` is assigned through a structural cast, and the divergence is silent until a runtime mismatch.
- **Root cause**: To keep the static import graph free of the `server-only` package (so the `tsx --test` runner can load the module — documented at operation.ts:1-6,45-50), the orchestrator re-declares the infra contracts structurally: `ChargeOutcome` is "Mirror of `ChargeResult`" (line 58), `AuthUser` is "structurally an `AppUser`" (line 53), `OperationLlm` re-states the LLM surface. These are deliberate decoupling seams, but they are maintained-by-hand parallel type definitions with no compile-time link to their originals.
- **Impact**: Moderate drift risk concentrated on the money path (`ChargeOutcome`). The decoupling is justified and the lazy-dynamic-import pattern is sound, but the duplicated type surface is a maintenance liability that will only grow as more routes adopt the orchestrator (finding 1) and depend on these mirrors being faithful.
- **Verification**: Read operation.ts:53-147 and the `defaultDeps` wiring (152-172): `getLlm`/`resolveUser` are assigned via `as`-casts (lines 161-162), `charge: chargeForOperation` relies on structural compatibility. No `import type { ChargeResult }` exists, so TS cannot flag a future field divergence. The mirroring is intentional and commented, hence medium not high.
- **Fix sketch**: Add a one-line type-assertion test (or `satisfies`) in a server-side test that imports the REAL `ChargeResult`/`AppUser` and asserts the local mirrors are assignable, so a future divergence is caught at compile/test time without pulling `server-only` into the orchestrator's static graph. No production-code change.

## 4. Docstring overstates centralization ("This module owns that pipeline once / ~500 lines duplicated")
- **Severity**: low
- **Category**: cleanup
- **File**: src/lib/ai/operation.ts:19-31
- **Scenario**: A new contributor reads the module header, believes all five charged AI routes already delegate here, and edits an invariant in one place expecting global effect — then is surprised when draft/rfe/categorize behavior is unchanged.
- **Root cause**: The header was written in the present tense for the END state of the ADR-0004 migration ("The five AI endpoints … all hand-implement the SAME pipeline … This module owns that pipeline once, so a route becomes a declarative AiOperationSpec"), but the migration stopped at 2/5 (finding 1). The prose now describes an aspiration as if it were fact.
- **Impact**: Cosmetic/documentation, but actively misleading on the exact point a maintainer most needs accuracy. Reinforces the false assumption behind the drift in finding 2.
- **Verification**: Compare operation.ts:21-31 ("The five AI endpoints … all hand-implement … This module owns that pipeline once") against the grep result of 2 adopting routes. The claim is currently false for draft, rfe, and evidence/categorize.
- **Fix sketch**: Reword to the true state, e.g. "Two routes (`guidance`, `qualify`) delegate here today; `draft`, `rfe`, and `evidence/categorize` still hand-roll this pipeline and are pending migration (ADR-0004, tasks 5-6 of 6)." Update (or remove) when finding 1 is resolved. Pure comment edit.
