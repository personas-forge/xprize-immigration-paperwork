## 1. Eval-harness engine mirror has drifted from the real client (stale "keep in sync" copy)
- **Severity**: high
- **Category**: duplication
- **File**: scripts/llm-eval/engine.ts:24-96 (mirrors src/lib/llm/client.ts:35-164)
- **Scenario**: `scripts/llm-eval/engine.ts` is a self-described "faithful mirror" of `client.ts` ("Keep in sync with `src/lib/llm/client.ts`") that re-implements the `Llm` interface, `GenerateOptions`, `getLlm()`, the gemini client, `claudeClient`, and `runClaudeCli` — only dropping the `server-only` import. It is the engine actually exercised by the eval harness (`run.ts:51`, `smoke.ts:10`). The two have diverged: (a) the mirror's `GenerateOptions` omits `temperature` (client.ts:43), so eval can never reproduce the `temperature: 0` determinism the `qualify` route relies on; (b) the mirror's gemini client builds `generationConfig` differently — `{ generationConfig: { responseMimeType } }` only when `json`, never setting `temperature` (engine.ts:50-55 vs client.ts:89-95); (c) the mirror has NO `LightTrack` tracking and NO `withGuards` wrapping, so the eval path skips the output-guard the production path always applies; (d) `runClaudeCli` in the mirror lacks the `child.stdin.on("error", reject)` hang-guard added to client.ts:155.
- **Root cause**: The `server-only` import in `client.ts` is unresolvable under a plain `tsx` run, so the wrapper body was copy-pasted instead of being factored into a `server-only`-free module that both `client.ts` and the harness import (the pattern `config.ts`/`json.ts`/`label.ts` already follow, and which `engine.ts` already reuses for `resolveEngine`).
- **Impact**: The eval harness validates a materially different client than production: no guard pass-rate, no temperature, divergent generationConfig, and a CLI path that can hang where prod no longer does. Bug fixes to `client.ts` (the stdin hang-guard was one) silently fail to reach the harness. "Keep in sync" by hand is already broken.
- **Verification**: Read both files end-to-end. `GenerateOptions` in engine.ts:24-27 has only `{ json, tier }`; client.ts:35-44 adds `temperature`. engine.ts has zero `LightTrack`/`withGuards` references (confirmed by grep over `scripts/`); client.ts:64-65 wraps both engines in `withGuards`. `child.stdin.on("error", ...)` present in client.ts:155, absent in engine.ts. Harness imports confirmed at `scripts/llm-eval/run.ts:51` and `smoke.ts:10`.
- **Fix sketch**: Extract the wrapper body (the `Llm` interface, `GenerateOptions`, `geminiClient`/`claudeClient`/`runClaudeCli`, and a `buildLlm(engine)` that does NOT call `resolveEngine`) into a `server-only`-free module — e.g. `src/lib/llm/engine.ts` — and have `client.ts` add only `import "server-only"` + the `getLlm()` entry that calls `resolveEngine`. The eval harness imports the shared module directly. This collapses the two `runClaudeCli`/gemini-client copies into one and removes the manual-sync hazard. Decide deliberately whether the harness should also exercise `withGuards`/LightTrack (it probably should).

## 2. JSON-coercion (`toSection` + `sections`-array parse) belongs next to `extractJson` in the LLM layer
- **Severity**: medium
- **Category**: duplication
- **File**: src/lib/llm/json.ts:9 (the missing shared seam) ; consumers src/features/drafting/drafting.ts:194-219, src/features/rfe/rfe.ts:152-177, src/features/drafting/saveRecovery.ts:61-68
- **Scenario**: `extractJson` (`@/lib/llm/json`) is the deliberately-shared tolerant-parse primitive — its own doc comment says it is "Shared by every JSON-returning AI feature (qualification, drafting, rfe, evidence)". But the layer that turns `extractJson`'s `unknown` into a typed `DraftSection` is NOT shared: `toSection(value): DraftSection | null` is byte-identical in drafting.ts:194, rfe.ts:152, and saveRecovery.ts:61, and the `extractJson → check .sections is array → .map(toSection).filter(...)` loop is identical between drafting.ts:209-219 and rfe.ts:167-177. So the model-output coercion is half-consolidated: the raw-JSON step lives in `@/lib/llm/json`, the shape-coercion step is triplicated in features.
- **Root cause**: When the JSON primitive was extracted (CHANGELOG: "`extractJson` ... consolidated into `@/lib/llm/json`"), only the brace-balancing step was lifted; the section-shape coercion that always immediately follows it was left behind in each feature.
- **Impact**: A change to section normalization (trimming, a max-length cap, a new field) must be made in three places or they drift. This is the natural second half of the `extractJson` consolidation and the LLM-layer home for it. (NOTE: the same triplication is reported from the feature side in `rfe-drafting.md` item #2 — this entry frames it as "the coercion belongs beside `extractJson`", not a second independent finding; fix once.)
- **Verification**: `grep -rn "function toSection"` → exactly drafting.ts, rfe.ts, saveRecovery.ts; bodies are byte-identical. `extractJson` importers: qualification.ts:21, drafting.ts:21, rfe.ts:21, evidence.ts:23 (matches its doc claim). Loop compared directly between drafting.ts:209-219 and rfe.ts:167-177.
- **Fix sketch**: Add `toSection(value): DraftSection | null` and `tryParseSections(text): DraftSection[] | null` beside `extractJson`. `DraftSection`'s home is `@/features/drafting`, so to avoid a feature→lib import cycle either relocate the `DraftSection` type to `@/lib/llm/sections` (a new pure module) or keep the helper generic over `{ heading; body }`. Have drafting/rfe/saveRecovery import it.

## 3. `guard()`'s rule surface is almost entirely unexercised (vendored, so document — don't prune)
- **Severity**: medium
- **Category**: dead-code
- **File**: src/lib/lighttrack.ts:102-205 (rules) ; only consumer src/lib/llm/guards.ts:23
- **Scenario**: The only `GuardRules` instance in the whole app is `LLM_OUTPUT_GUARD = { minWords: 1, maxChars: 50000 }` (guards.ts:23), applied via `withGuards`. Every other `guard()` capability — `json`, `jsonKeys`, `maxWords`, `mustInclude`, `mustMatch`, `mustNotMatch`, `noPII` plus the four `PII_PATTERNS` (lighttrack.ts:126-131) — has zero callers in `src/`. So roughly 80% of the deterministic guard engine (and the PII regexes) is reachable code that nothing reaches. The richer rule that the JSON routes would benefit from (`{ json: true, jsonKeys: [...] }`) is exactly what's NOT used — those routes instead rely on the spec's `guard(raw)` parser in `lib/ai/operation.ts:272` returning null.
- **Root cause**: `lighttrack.ts` is VENDORED ("keep in sync with upstream; don't edit locally", lighttrack.ts:1-3) — `guard()`/`GuardRules` is upstream's full feature set, intentionally carried whole. The app only opted into the length checks. This is NOT removable dead code in the usual sense; pruning it would diverge from upstream.
- **Impact**: Misleading surface — a reader (or a future guard rule) sees a full PII/JSON/format validator that has never run those branches in this app, and may assume the routes are PII-guarded when they are not. The risk is "false sense of coverage", not maintenance cost.
- **Verification**: Grep over `src/` (excluding lighttrack.ts) for `jsonKeys|mustMatch|mustInclude|noPII|mustNotMatch|maxWords` → no matches; only `minWords`/`maxChars` appear, both in guards.ts:23. `guard()` itself is called only inside `trackGuard` (lighttrack.ts:283); `trackGuard` is called only by `withGuards` (guards.ts:34). Vendoring banner confirmed at lighttrack.ts:1-3.
- **Fix sketch**: Do NOT delete (vendored). Instead, if PII/JSON guarding is desired, OPT IN by tightening `LLM_OUTPUT_GUARD` or adding a per-route guard rule (e.g. `{ json: true, jsonKeys: ["sections"] }` for draft/rfe) — that turns dormant code into used code. If it's confirmed undesired, add a one-line comment on `LLM_OUTPUT_GUARD` noting the app deliberately uses only length checks, so the unused-branch gap is intentional and documented. No breaking-change risk either way.

## 4. `extractJson` doc comment and one test name contradict the actual (balanced-brace) behavior
- **Severity**: low
- **Category**: cleanup
- **File**: src/lib/llm/json.ts:9 (and inline comment 14-17) ; src/lib/llm/json.test.ts:19
- **Scenario**: `extractJson` finds the brace that *balances* the first `{` (json.ts:18-41, with an explicit comment at 14-17 explaining it deliberately does NOT use `lastIndexOf("}")` because that over-grabs on a trailing second object). But the test that pins this behavior is titled `"extractJson: takes the outermost braces (first { to last })"` (json.test.ts:19) — "first { to last }" is precisely the `lastIndexOf` strategy the implementation rejects. The test assertion itself (`{"a":{"b":2}}` → nested) is correct and passes under both strategies, so the title is just wrong, but it documents the opposite of what the code guarantees.
- **Root cause**: The balanced-brace rewrite (CHANGELOG:95: "`extractJson` balances braces instead of grabbing first-to-last") updated the implementation and its inline comment but left the old test title describing the pre-rewrite "first-to-last" behavior.
- **Impact**: A maintainer reading the test name could "simplify" `extractJson` back to `slice(indexOf("{"), lastIndexOf("}")+1)` believing the test sanctions it — reintroducing the `{"a":1} note {"b":2}` over-grab bug the rewrite fixed. Purely a documentation/test-naming hazard; behavior is correct.
- **Verification**: Read json.ts (balanced-depth loop, lines 18-41) and json.test.ts:19-21; the title says "first { to last }", the code comment (14-17) explicitly argues against `lastIndexOf("}")`. CHANGELOG:95 confirms the intended behavior is brace-balancing.
- **Fix sketch**: Rename the test to e.g. `"extractJson: keeps nested objects (balances braces)"` and add a regression case asserting the over-grab is avoided: `extractJson('{"a":1} note {"b":2}')` should yield `{ a: 1 }`, not null/garbage. Optionally tighten `extractJson`'s JSDoc to say "balanced first object" rather than leaving the contract implicit.
