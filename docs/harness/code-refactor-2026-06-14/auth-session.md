## 1. Owner-or-attorney case gate re-implemented inline in the case-detail page
- **Severity**: high
- **Category**: duplication
- **File**: src/app/dashboard/cases/[id]/page.tsx:42-48
- **Scenario**: The case-detail page resolves access with a hand-rolled copy of the exact fail-closed gate that `resolveCase` (src/lib/data/adapters/access.ts) and `authorizeRoute` (src/lib/auth/authorizeRoute.ts) were created to centralize: `const attorney = isConfiguredAttorney(user.email); const owned = await getCaseForUser(user.id, id); const stored = owned ?? (attorney ? await getCaseAnyOwner(id) : null);`.
- **Root cause**: ADR-0010 (`access.ts` header) explicitly states this resolution "was copy-pasted across rfe/route.ts, evidence/categorize/route.ts, draft/route.ts and review/actions.ts" and now "exists exactly ONCE" in `resolveCase`. The API routes and `review/actions.ts` were migrated to the adapter; this server-component page is the one straggler that still inlines the gate. Verified the helper is otherwise the single source: only this file performs the `getCaseForUser(...) ?? (... getCaseAnyOwner(...))` expression outside `access.ts`/`authorizeRoute.ts`.
- **Impact**: The whole security argument of the adapter layer ("the cross-tenant `isConfiguredAttorney` check can never be forgotten") is undermined by a fifth inline copy. It is the only cross-tenant PII-egress gate not behind the helper, so a future edit here (e.g. swapping `isConfiguredAttorney` for the permissive `isAttorney`) would silently reopen the HIGH IDOR class the team closed — exactly the regression `resolveCase` is designed to make impossible.
- **Verification**: Grep of `getCaseAnyOwner`/`getCaseForUser`/`isConfiguredAttorney` across `src` shows every other case-scoped read routes through `petitions.resolveCase` (the adapter) or `authorizeRoute`; this page imports the raw `getCaseForUser`/`getCaseAnyOwner` from `@/lib/data/petitions` and combines them itself. The page already runs in the Node runtime (`requireOnboardedUser()`), so the adapter (server-only, lazy-imported) is usable here.
- **Fix sketch**: Replace lines 42-48 with `const gate = await petitions.resolveCase({ userId: user.id, email: user.email }, id);` then map `gate` (`ok`→`stored`, `not_found`/`forbidden`→`notFound()`, `unconfigured`/`store_error`→appropriate handling). Derive `isOwner` from a separate owner-only resolve (`{ userId: user.id, email: null }`) — mirroring the pattern `review/actions.ts:80` already uses — since `StoredCase` carries no owner field. Do NOT change the policy: still owner OR configured-attorney, fail-closed. This is behavior-preserving and removes the last inline copy.

## 2. Dead `setCaseStatus` data-layer wrapper — superseded by `transitionCase`
- **Severity**: medium
- **Category**: dead-code
- **File**: src/lib/data/reviews.ts:61-69
- **Scenario**: `reviews.setCaseStatus(caseId, status, opts)` wraps `Store.setCaseStatus`, but no production code path calls the wrapper. All case status changes go through the atomic compare-and-set `transitionCase` (review/actions.ts: submitForReview, attorneyRequestChanges, attorneySignAndFile, attorneyRecordDecision), which was introduced specifically to fix double-submit / audit-log-desync that the non-atomic `setCaseStatus` allowed.
- **Root cause**: `transitionCase` superseded `setCaseStatus` for the review workflow; the older one-shot setter was left in the data layer (and the Store interface) without a remaining caller.
- **Impact**: Dead but security-relevant surface — `reviews.setCaseStatus` performs NO ownership/attorney gate (its own header notes callers must gate). Leaving an ungated, status-mutating, no-arg-validation function exported invites a future caller to bypass the `transitionCase` compare-and-set guard and illegally re-open a Filed case. Removing it shrinks the mutation surface.
- **Verification**: Grep for `setCaseStatus` across `src`: the only references are the wrapper itself, the `Store` interface declaration (store.ts:211), both driver impls (firestore-store.ts:314, pglite-store.ts:460), the events bus dispatch (`store-events.ts:41/47`, switches on method name), and tests. No `import { setCaseStatus }` and no `reviews.setCaseStatus(` call site in any route/page/action. (Contrast: `transitionCase`, `addReviewEvent`, `getReviewEvents` all have live callers.)
- **Fix sketch**: Remove the `reviews.setCaseStatus` data-layer wrapper. The `Store.setCaseStatus` method + driver impls are reachable via the generic `withDomainEvents` dispatch and the `CaseStatusChanged` event mapping, so the Store-level method is a larger/riskier removal — leave it OR retire it together with the events mapping in a separate, test-backed change. Confirm no dynamic dispatch references the data wrapper before deleting (none found).

## 3. Firebase-admin `initializeApp` block duplicated verbatim across two modules
- **Severity**: low
- **Category**: duplication
- **File**: src/lib/firestore/admin.ts:14-20 (and src/lib/firebase/admin.ts:13-19)
- **Scenario**: Both `adminAuth()` (firebase/admin.ts) and `adminDb()` (firestore/admin.ts) contain the identical guarded init: `if (!getApps().length) initializeApp({ credential: applicationDefault(), projectId: process.env.NEXT_PUBLIC_FIREBASE_PROJECT_ID });`. firestore/admin.ts's own header even states it "shares the default firebase-admin app with @/lib/firebase/admin".
- **Root cause**: Two consumers of the same default Admin app each lazily ensure init independently, copy-pasting the bootstrap rather than sharing one `adminApp()` initializer.
- **Impact**: Correctness is fine (both guard on `getApps().length`, so the second is a no-op and they share one app) — this is maintainability only. Drift risk: the inline `projectId` uses raw `process.env.NEXT_PUBLIC_FIREBASE_PROJECT_ID`, whereas `firestoreProjectId()` in db/config.ts is the richer resolver (`FIRESTORE_PROJECT_ID || NEXT_PUBLIC_FIREBASE_PROJECT_ID || GOOGLE_CLOUD_PROJECT`); the two admin entrypoints could silently diverge from the rest of the app on project-id resolution.
- **Verification**: Read both files; the init blocks are byte-identical apart from `getAuth()` vs `getFirestore()`. Grep confirms these are the only two `initializeApp` call sites for firebase-admin in `src`.
- **Fix sketch**: Extract a tiny shared `ensureAdminApp()` (e.g. in firebase/admin.ts) that owns the single guarded `initializeApp` and have both `adminAuth()` and `adminDb()` call it; have `getFirestore()`/`getAuth()` read from that app. Do NOT change credential resolution (`applicationDefault()`) — auth behavior must stay identical. Optionally align `projectId` on `firestoreProjectId()` in the same change, but verify that doesn't alter the resolved project in any deployed env first.

## 4. `addReviewNote` resolves the gate twice (owner-only resolveCase + separate attorney check)
- **Severity**: low
- **Category**: cleanup
- **File**: src/features/review/actions.ts:80-82
- **Scenario**: `addReviewNote` calls `petitions.resolveCase({ userId, email: null }, caseId)` (owner-only, by passing `email: null`) for `owned`, then separately calls `isConfiguredAttorney(user.email)` for `attorney`, and gates on `!owned && !attorney`. This re-derives the attorney leg of the gate by hand instead of letting `resolveCase` do it with `email` supplied.
- **Root cause**: The author needed `owned` as a boolean to drive `authorRole` attribution (`owned ? "applicant" : "attorney"`) because `StoredCase` has no owner field, so they ran an owner-only resolve and bolted the attorney check on the side — partially re-implementing the very owner-or-attorney logic `resolveCase` centralizes.
- **Impact**: Minor. Behavior is correct and fail-closed (the inline `isConfiguredAttorney` matches the helper's attorney leg), but it's a second hand-rolled instance of "owner OR configured-attorney" in the same family as finding #1, and the comment block (lines 74-79) explicitly acknowledges it mirrors the prior split "byte-for-byte". More logic to keep in sync than necessary.
- **Verification**: Read actions.ts:65-91. `owned` is used only as a boolean (`.ok`) and to pick `authorRole`; `attorney` is used only in the `!owned && !attorney` guard. The owner-only resolve + separate `isConfiguredAttorney` reproduces `resolveCase`'s decision.
- **Fix sketch**: Keep the owner-only resolve solely for `authorRole` attribution (it genuinely needs the owner/non-owner distinction), but consider deriving the authorization decision from a single `resolveCase({ userId, email: user.email }, caseId)` and reserving the owner-only resolve purely for the role label — or accept as-is given the explicit byte-for-byte preservation rationale. Lowest priority; do not touch unless #1 is being addressed in the same pass. No security change.
