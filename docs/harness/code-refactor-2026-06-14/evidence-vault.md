## 1. `categorize` route hand-rolls the same auth → rate-limit → charge → LLM-or-mock-fallback preamble as `draft` and `rfe`
- **Severity**: high
- **Category**: duplication
- **File**: src/app/api/evidence/categorize/route.ts:35-125 (vs src/app/api/draft/route.ts:52-274 and src/app/api/rfe/route.ts:36-167)
- **Scenario**: All three charged AI routes repeat a near-identical body: (a) `authorizeRoute(request, {...})` up front to read caseId from a clone before `request.json()`; (b) the `if (isRateLimitEnabled())` block with `checkRateLimit(rateLimitKey(request, "<op>", ...), RATE_LIMITS.<op>)` returning the same 429 `{ error: "rate_limited", retryAfterSec, disclaimer }` + `Retry-After` header; (c) `const requestId = randomUUID(); const charged = await chargeForOperation("<op>", requestId)` with the identical `unauthenticated → 401` / else `402 { error: "insufficient_tokens", cost, balance, disclaimer }` branch; (d) the `const llm = getLlm(); if (!llm) mock else try { generate(); tryParse → keep : await charged.reclaim() + mock } catch { reclaim + mock }` shape, byte-for-byte in structure.
- **Root cause**: No shared route harness. Each route was authored by copy-paste from the previous one; the only per-route variation is the operation name, the prompt/parse functions, and (for categorize) that persistence is best-effort rather than gating the response.
- **Impact**: The three blocks must be kept in lock-step by hand. The token-reclaim-on-unusable-output invariant (the money-path correctness the comments stress) lives in three places; a fix to one (e.g. the 429 envelope, or forgetting `reclaim()`) silently diverges in the others. High-value because it is the charged, security-sensitive surface, and the divergence is already visible: rfe wraps its criteria read in try/catch returning 503 while draft routes the same failure through `toErrorResponse`.
- **Verification**: Read of all three route files confirms the repeated blocks. Grep for `chargeForOperation` / `checkRateLimit` / `charged.reclaim` shows the identical pattern in categorize, draft, rfe (and qualify for the charge half). No shared helper module exists (`@/lib/auth/authorizeRoute` and `@/lib/tokens/guard` are the only shared pieces; the orchestration around them is not extracted).
- **Fix sketch**: Extract a `withChargedAiRoute(request, { op, requiresCase, requiresAttorney })` helper (or a thin `runCharged(op, requestId, () => llmAttempt)` that owns the reclaim-on-null/throw branch and the 401/402 envelope) into `@/lib/llm` or a new `@/lib/api/route-kit`. Routes then supply only prompt + parse + persistence. Do NOT alter the per-route `requiresAttorney` value (categorize/rfe true, draft false) — that is a real access-policy difference, not duplication.

## 2. `categorize` route persists through the raw data layer + `authorizeRoute`, while the vault's own server actions gate through `EvidenceAdapter` — two gate paths for one vault
- **Severity**: high
- **Category**: structure
- **File**: src/app/api/evidence/categorize/route.ts:21,44-50,131-143 (raw `addCaseDocument` + `authorizeRoute` gate) vs src/features/evidence/actions.ts:27,40,53 (adapter `evidence.removeDocument`/`refileDocument` via `resolveCase`)
- **Scenario**: Within this one context, the add path and the remove/refile paths gate the SAME owner-or-attorney rule through two different mechanisms. `actions.ts` routes mutations through `EvidenceAdapter`, whose `resolveCase` seam (ADR-0010) re-checks `isConfiguredAttorney` immediately before touching the store. The categorize route instead imports `addCaseDocument` directly from `@/lib/data/evidence` (which explicitly enforces nothing) and relies on a separate, earlier `authorizeRoute(..., { requiresAttorney: true })` call for the gate. `EvidenceAdapter.addDocument` (adapters/evidence.ts:81) exists, is tested, and would centralize exactly this — but has zero production callers.
- **Root cause**: Partial ADR-0010 adoption: the remove/refile actions were migrated to the adapter; the categorize route (written against the route-level `authorizeRoute` guard) was not.
- **Impact**: The access rule has two homes inside a single feature. Currently correct — `authorizeRoute` with `requiresAttorney:true` and `resolveCase` evaluate the same `isConfiguredAttorney` fallback — but any future tightening of the attorney rule must be remembered in both, and a reviewer reading `actions.ts` reasonably assumes the whole vault goes through the adapter seam, masking that the WRITE path does not. (Cross-ref: docs/harness/code-refactor-2026-06-14/data-adapter.md finding #1 catalogs the unadopted adapter methods from the adapter-layer side; this finding is the evidence-context consequence — the dual-gate split — and should be resolved together.)
- **Verification**: Repo-wide grep for `.addDocument(` / `.getDocuments(` outside `src/lib/data/adapters/**` and `*.test.ts` returns zero hits (confirmed against data-adapter.md). categorize/route.ts:21 imports the raw fn; actions.ts:27 imports the adapter `evidence`. Both gates resolve `isConfiguredAttorney` (authorizeRoute path + access.ts resolveCase) — behaviour matches today, so this is structural duplication risk, not an active bug.
- **Fix sketch**: Migrate the categorize persistence to `evidence.addDocument(access, {...})`, deriving `access` from the already-resolved `auth` (it has the user + email), and drop the raw `addCaseDocument` import. Keep persistence best-effort (treat any non-`ok` AdapterResult as "no document", as the route does today) — do NOT promote a store/forbidden outcome to a hard error, and do NOT weaken `requiresAttorney`.

## 3. `src/lib/data/documents.ts` is a dead, superseded mock data layer (zero consumers)
- **Severity**: medium
- **Category**: dead-code
- **File**: src/lib/data/documents.ts:9-26 (the `documents` fixture + `getDocuments`), re-exported at src/lib/data/index.ts:18
- **Scenario**: `documents.ts` holds a hardcoded 7-row `CaseDocument[]` fixture and a `getDocuments()` accessor, exported via the `@/lib/data` barrel. Its header comment claims it "Feeds the case checklist export," but nothing imports `getDocuments`. The live evidence vault reads through `@/lib/data/evidence.ts` (`getCaseDocuments`, a real Store query) at the case-detail page (cases/[id]/page.tsx:13,56). The checklist export (`features/case-file/export.ts` `checklistToCsv`) takes a `readonly CaseDocument[]` argument from its callers — it is NOT fed by `getDocuments()`.
- **Root cause**: `documents.ts` was the original in-memory fixture for the vault; it was superseded by the Store-backed `lib/data/evidence.ts` and never removed, leaving a stale module + barrel re-export.
- **Impact**: Dead module masquerading as live (the doc comment and barrel placement imply it backs the export). Misleads readers into thinking the vault has a fixture fallback; the hardcoded "Anya" sample data is also shipped in the bundle for any code that touches the `@/lib/data` barrel.
- **Verification**: Grep for `getDocuments` / `data/documents` across `src` (excluding tests) returns only the definition (documents.ts:24) and the barrel re-export (index.ts:18) — no importer. The three real `@/lib/data` consumers (CaseList.tsx:5, useCaseFileData.ts:4, FieldGuidancePanel.tsx:7) import `getCases` / `getCaseFacts`/`getCriteria`/`getOutstandingTasks`/`getPetitionExcerpt` / `getForms`, never `getDocuments`. The `CaseDocument` TYPE itself stays live (used by export.ts:1,54 and export.test.ts) so only this module is dead, not the type.
- **Fix sketch**: Delete `src/lib/data/documents.ts` and the `export { getDocuments } from "./documents";` line in index.ts. No other change needed — `CaseDocument` remains exported from `@/features/case-file/types`. Low breaking-change risk (no importers); confirm `tsc` after removing the barrel line.

## 4. Duplicate `DocumentView` / `StoredDocument` shape declared three times across the vault
- **Severity**: low
- **Category**: cleanup
- **File**: src/features/evidence/components/EvidenceVault.tsx:18-26 (`DocumentView`) vs src/lib/data/evidence.ts:18-26 (`StoredDocument`); the route re-asserts the field set inline at categorize/route.ts:133-139
- **Scenario**: `DocumentView` (client) and `StoredDocument` (data layer) are field-for-field identical — `{ id, name, criterion, exhibit, status, facts: string[], source }` — declared independently. The categorize route's `addCaseDocument({...})` call and the `EvidenceDeps.addCaseDocument` signature (adapters/evidence.ts:26-33) each re-spell the same insert shape. The client also relies on the API JSON matching `DocumentView` with no shared contract type.
- **Root cause**: The client component can't import the `server-only` `StoredDocument` (it would pull `import "server-only"`), so the shape was hand-duplicated rather than factored into a shared, server-free type module.
- **Impact**: Low / cosmetic. Three copies drift independently; adding a vault field (e.g. an upload timestamp) means editing the type, the component view, the route payload assembly, and the adapter dep signature with no compiler link between them. No correctness or security effect today.
- **Verification**: Read of EvidenceVault.tsx:18-26 and lib/data/evidence.ts:18-26 confirms identical members; categorize/route.ts:133-139 and adapters/evidence.ts:26-33 repeat the insert field set. `DocumentView` is referenced only within EvidenceVault.tsx (grep shows no external importer), so consolidation is contained.
- **Fix sketch**: Move the plain document shape into a server-free types module (e.g. `@/features/evidence/types.ts`), export `StoredDocument` from it, and have both `lib/data/evidence.ts` and `EvidenceVault.tsx` import it; `DocumentView = StoredDocument`. Leave the `server-only` boundary on the accessor functions, not the type. Optional given low impact.
