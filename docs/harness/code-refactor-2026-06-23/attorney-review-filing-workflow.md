## 1. Demo-receipt flag re-derived in the UI by regex instead of reading the stored boolean
- **Severity**: Critical
- **Category**: duplication
- **File**: src/features/review/components/ReviewPanel.tsx:77-79 (consumer) ; src/features/review/actions.ts:225,248 (producer)
- **Scenario**: `attorneySignAndFile` already computes the authoritative demo/genuine distinction once — `const isDemo = realReceipt === ""` (actions.ts:225) — and persists it structurally as `metadata: { receipt, demo: isDemo }` on the `filed` event (actions.ts:248), plus a human body string `"Recorded a DEMO filing (not actually filed with USCIS)."`. The ReviewPanel then RE-DERIVES the same fact a second way: `events.some((ev) => ev.kind === "filed" && /DEMO/i.test(ev.body))` (ReviewPanel.tsx:77-79), pattern-matching the prose body rather than reading `metadata.demo`. Grep: `DEMO|isDemo|receiptIsDemo|demo:` across src/ shows the boolean is written in actions.ts and the regex re-detection only in ReviewPanel.tsx — the structured field is never read back.
- **Root cause**: Two independent encodings of one boolean truth. The UI doesn't receive `metadata` on `ReviewEventView` (it carries only `id/authorRole/kind/body/when`), so the panel reaches for the one field it does have (the body copy) and reconstructs the flag by string-matching.
- **Impact**: Divergence-prone for a legally-sensitive label. The whole point of `receiptIsDemo` is to stop a demo receipt being mistaken for a genuine USCIS receipt — but any edit to the body copy (rewording, i18n/localization, dropping the literal word "DEMO", or sending the string through a translator) silently flips a real demo receipt to render as a genuine "USCIS receipt" with the success (green) styling. The source-of-truth boolean would be unaffected; the UI would be wrong. Two strings that must stay byte-aligned forever, with no test pinning them together.
- **Fix sketch**: Thread the authoritative signal through instead of re-detecting it. Either (a) add `demo: boolean` (from `metadata.demo`) to `ReviewEventView` and have the case-detail page map `ev.metadata?.demo` into it, then `receiptIsDemo = events.some(ev => ev.kind === "filed" && ev.demo)`; or (b) compute `receiptIsDemo` server-side in the page and pass it as a prop. Drop the `/DEMO/i` regex entirely.

## 2. The three USCIS decision values are hard-coded in two places (server allowlist vs. <select> options)
- **Severity**: High
- **Category**: duplication
- **File**: src/features/review/actions.ts:266 ; src/features/review/components/ReviewPanel.tsx:178-180
- **Scenario**: `attorneyRecordDecision` validates against an inline server-side allowlist `["Approved", "RFE issued", "Denied"]` (actions.ts:266), and the ReviewPanel renders the exact same three strings as literal `<option value="…">` entries (ReviewPanel.tsx:178-180). Grep `RFE issued|UscisDecision|DECISIONS` across src/ confirms these are the ONLY two occurrences — there is no shared constant; the list is copy-pasted between the form and its validator.
- **Root cause**: No single source of truth for the closed set of decision outcomes. The `<select>` and its server allowlist evolved as two literal lists rather than one exported `readonly` tuple.
- **Impact**: Classic UI/validator drift. Add or rename an option in the panel (e.g. "RFE issued" → "RFE") and the server silently rejects it with "Unrecognized decision"; tighten the server list and a stale option still renders, dead on submit. Because only "Approved" is terminal (drives the case→"Approved" transition, actions.ts:274), a divergence here mis-routes the case lifecycle, not just a label.
- **Fix sketch**: Export `export const USCIS_DECISIONS = ["Approved", "RFE issued", "Denied"] as const;` from a shared module (e.g. alongside `CaseStatus` in case-file/types.ts or a small review constants file). Server: `if (!USCIS_DECISIONS.includes(decision as …))`. Panel: `{USCIS_DECISIONS.map((d) => <option key={d} value={d}>{d}</option>)}`.

## 3. Owner-only "resolve with email:null then derive isOwner" pattern is repeated three times
- **Severity**: Medium
- **Category**: duplication
- **File**: src/features/review/actions.ts:109 ; src/features/review/actions.ts:138 ; src/app/dashboard/cases/[id]/page.tsx:58
- **Scenario**: The owner-only gate `petitions.resolveCase({ userId: user.id, email: null }, caseId)` (email deliberately nulled so the configured-attorney fallback can't fire) appears verbatim in `submitForReview` (actions.ts:109), in `addReviewNote`'s ownership branch where its `.ok` becomes `owned` (actions.ts:138), and again in the case-detail page to compute `isOwner` (page.tsx:58). Grep `resolveCase({ userId: user.id, email: null }, caseId)` returns the two action sites; the page uses the same shape with `id`. Each carries its own multi-line comment re-explaining the `email: null ⇒ owner-only` trick, and page.tsx:56-57 explicitly notes it's copying "the pattern review/actions.ts uses."
- **Root cause**: The "is this caller the owner?" question has no named helper, so every caller re-expresses the adapter incantation plus the comment that justifies it. (Note: this is NOT the intentional attorney-vs-owner gate separation called out in grounding — it's the owner-only resolve specifically, duplicated.)
- **Impact**: The semantics live in an easily-mistyped argument (`email: null`). A future edit that passes `user.email` at one of the three sites would silently widen an owner-only check to admit configured attorneys, reopening exactly the cross-tenant class the comments warn about — and the reviewer must re-derive the subtlety three times. The `owner-only-gate.test.ts` contract pins the adapter behavior but not that all three callers keep passing `email: null`.
- **Fix sketch**: Add a tiny named helper, e.g. `isCaseOwner(userId, caseId): Promise<boolean>` (or `resolveOwnerOnly(...)`) in the adapter/access layer that hard-codes `email: null`, and call it from all three sites. The intent ("owner-only") becomes the function name; the `email: null` trick and its paragraph of justification live in one place.

## 4. CaseStatus union re-listed as an inline literal array in saved-cases.ts
- **Severity**: Medium
- **Category**: duplication
- **File**: src/lib/data/saved-cases.ts:39 ; canonical: src/features/case-file/types.ts:27-32
- **Scenario**: `getCases()` validates a stored status with an inline tuple `(["Intake", "Drafting", "Attorney Review", "Filed", "Approved"] as const).includes(c.status as never)` (saved-cases.ts:39). The identical five members are the canonical `CaseStatus` union (types.ts:27-32), whose own docstring says it must "stay in lockstep with the data layer." Grep `Intake.*Drafting.*Attorney Review.*Filed.*Approved` finds this saved-cases.ts copy; `CaseStatus` greps show CaseList.tsx and usePersistentQuery.ts also keep parallel `STATUSES` arrays — but those are UI filter lists that legitimately prepend "all"; saved-cases.ts is a pure runtime guard for the exact union and is the cleanest candidate to derive.
- **Root cause**: No runtime companion to the `CaseStatus` type, so a third literal copy was written to do the membership check, decoupled from the type it's guarding.
- **Impact**: Add a sixth lifecycle status (e.g. "Withdrawn") to `CaseStatus` and this guard silently coerces every such persisted case to `"Intake"` (the fallback at the end of the ternary), because the inline array wasn't updated. The type compiles green; the data is wrong. The duplication defeats the type's stated "lockstep" intent.
- **Fix sketch**: Export a single `export const CASE_STATUSES = ["Intake","Drafting","Attorney Review","Filed","Approved"] as const;` in types.ts and `export type CaseStatus = (typeof CASE_STATUSES)[number];`. saved-cases.ts then does `(CASE_STATUSES as readonly string[]).includes(c.status)` — one list, type and guard derived from it.

## 5. attorneyRecordDecision bypasses the formField() helper it sits next to
- **Severity**: Low
- **Category**: cleanup
- **File**: src/features/review/actions.ts:262 (vs helper at actions.ts:59-61)
- **Scenario**: `formField(formData, name, max)` was added (actions.ts:59-61) expressly to centralize untrusted-FormData coercion ("coerce to string, trim, and cap length … so every review-note/feedback field shares it"), and `addReviewNote` / `attorneyRequestChanges` / `attorneySignAndFile` all route through it. But `attorneyRecordDecision` reads the decision with a raw `String(formData.get("decision") ?? "Approved")` (actions.ts:262), not the helper. Grep `String\(formData\.get` in this file returns exactly the helper body (line 60) and this one open-coded site (line 262).
- **Root cause**: The decision field wants a different default ("Approved") and no length-cap, so the author inlined the read instead of extending the helper — leaving one of four FormData reads off the shared path.
- **Impact**: Cosmetic/consistency only (the value is immediately allowlist-checked at line 266, so there's no correctness gap). But it's a small reader trap: the module advertises "every field shares" the helper, yet one field doesn't, so a future change to coercion policy (e.g. NFKC-normalize, strip control chars) lands on three of four fields. Minor inconsistency, not a bug.
- **Fix sketch**: Either read it via the helper with no cap — `const decision = formField(formData, "decision") || "Approved";` (note: this trims, which is harmless before the exact-match allowlist) — or leave as-is and drop the helper's "every … field shares it" claim so the doc matches reality. Lowest-risk: route it through `formField` for one coercion path.
