## 1. The owner/attorney case-resolve preamble is hand-copied across four AI specs
- **Severity**: High
- **Category**: duplication
- **File**: src/app/api/rfe/route.ts:62-89, src/features/drafting/draftOperation.ts:135-161, src/features/drafting/critiqueOperation.ts:79-107, src/features/rfe/forecastOperation.ts:74-112 (variant in src/app/api/evidence/categorize/route.ts:71-83)
- **Scenario**: Every spec that accepts a `caseId` opens `parse` with the SAME block: `const user = await resolveUser(); if (!user) return 401`, then `const access: CaseAccess = { userId: user.id, email: … }`, then `const gate = await petitions.resolveCase(access, caseId)`, then `if (!gate.ok) { if (gate.error.kind === "forbidden" || gate.error.kind === "not_found") return 403 "You don't have access to this case."; return toErrorResponse(gate.error) }`. Greps `gate.error.kind === "forbidden" || gate.error.kind === "not_found"` (4 AI-spec hits + 1 page) and `"You don't have access to this case."` (the rfe/draft/critique/forecast specs) confirm the block is copied verbatim four times. The 403/401 message strings, the kind-check, and the `toErrorResponse` fall-through are character-identical in rfe/draft/critique; forecast/critique only differ by using a literal `503 "… unavailable."` instead of `toErrorResponse` for the store-fault leg.
- **Root cause**: When `parse` was made the home for "owner gates that must run before charge" (its doc comment), each spec author re-implemented the resolve+gate ladder inline. There is no shared helper for "resolve a case for this access, or return the canonical 401/403/store-fault response."
- **Impact**: Four copies of an access-control decision drift independently — exactly the seam the adapter exists to centralize. The forecast/critique divergence (literal 503 vs `toErrorResponse`) is already a live inconsistency: a store fault surfaces a different body/shape on `/api/rfe` than on `/api/rfe/forecast` for the same underlying failure. Any future change to the gate contract (e.g. distinguishing `not_found` from `forbidden`, or adding an audit log) must be made in four places or one silently diverges — an authorization divergence bug, the highest-cost class for this cleanup lane.
- **Fix sketch**: Add a `resolveCaseForParse(access, caseId): Promise<{ ok: true; case: … } | { ok: false; response: NextResponse }>` helper next to `CaseAccess`/`toErrorResponse` (e.g. `src/lib/data/adapters/access.ts`) that owns the resolveUser→401, resolveCase→403, store-fault→typed ladder once. Each spec's `parse` then becomes `const r = await resolveCaseForParse(access, caseId); if (!r.ok) return r;`. Settle the 503-vs-`toErrorResponse` question in that one place.

## 2. `ParseContext.request` is a public field no consumer reads
- **Severity**: Medium
- **Category**: dead-code
- **File**: src/lib/ai/operation.ts:81-86 (declared), :278 (passed)
- **Scenario**: `ParseContext` exposes `request: Request`, and the orchestrator passes it at line 278 (`spec.parse({ request, body, resolveUser })`). Grep of every `parse:` implementation shows all nine destructure only `{ body }` or `{ body, resolveUser }` (operation.test.ts:62, forecastOperation.ts:65, draftOperation.ts:117, critiqueOperation.ts:57, guidance:47, categorize:50, rfe:53, qualify:43, best-path:31) — NONE reads `request`. Grep `ctx.request`/`\.request\b` inside the spec files returns nothing. The orchestrator itself reads headers (`readIdempotencyKey(request)`) and builds the rate-limit key (`d.rateLimit.key(request, …)`) from its OWN `request` param, not via the context, so the field is purely vestigial on the spec-facing API.
- **Root cause**: `request` was put on `ParseContext` defensively when the interface was designed (a parse might want the raw URL/headers), but the actual cross-cutting needs — idempotency header, rate-limit key — were absorbed by the orchestrator, so no spec ever needed it.
- **Impact**: A public interface field that implies parse hooks may/should read the raw request, when in practice the design deliberately keeps header/IP concerns in the orchestrator. It widens the documented surface and invites a spec to start reading headers ad hoc (re-introducing the per-route boilerplate the orchestrator removed). Dead, but low blast radius.
- **Fix sketch**: Drop `request` from `ParseContext` and from the call site at :278 (no consumer change needed). If a future multipart/upload op genuinely needs the raw request, re-add it with that justification.

## 3. `requiresImages` is a wired-but-unexercised extension point (no spec sets it)
- **Severity**: Low
- **Category**: dead-code
- **File**: src/lib/ai/operation.ts:116 (spec field), :172 & :341 (plumbed to `getLlm`)
- **Scenario**: `spec.requiresImages` is threaded into `d.getLlm({ requiresImages: spec.requiresImages })` and on into `selectEngine` (engines.ts:53-55, which excludes the Claude CLI for image ops). But grep `requiresImages:\s*true` across `src/` finds it set NOWHERE — only in `client.ts`/`engines.ts` doc comments and the orchestrator's own plumbing. The categorize spec even carries a TODO at categorize/route.ts:87 ("when binary PDF/image OCR is added, set requiresImages"). So today the whole flag is always `undefined`, the multimodal branch is never taken, and no test fixture sets it. (Per the grounding, `onBlocked`/`Idempotency-Key`/`adjudicate` are intentional extension points AND each has a real spec/test using it — `onBlocked` is used by guidance.ts:96 — so they are NOT flagged. `requiresImages` is the one extension point with zero current users.)
- **Root cause**: The image-routing seam was built ahead of the first multimodal operation (OCR), which has not shipped — the categorize TODO is the intended future caller.
- **Impact**: Minimal and arguably justified as forward-design — flagged only for the CERTAINTY ledger. The cost is a small always-`undefined` code path plus a `getLlm` arg that no run exercises, so a regression in the image-exclusion routing (engines.ts:55) would not be caught by any current test.
- **Fix sketch**: Keep it (it is cheap and the OCR caller is concretely planned), but either (a) land the categorize `requiresImages: true` when OCR ships, or (b) add a single orchestrator test that passes `requiresImages: true` and asserts it reaches `getLlm`, so the seam is covered before its first real use. Do not delete — that would just have to be rebuilt for OCR.

## 4. `caseId` extraction idiom repeated 4× with a real trim inconsistency
- **Severity**: Low
- **Category**: duplication
- **File**: src/features/rfe/forecastOperation.ts:67-70, src/app/api/rfe/route.ts:55-58, src/features/drafting/draftOperation.ts:120-123, src/features/drafting/critiqueOperation.ts:69-72 vs. src/app/api/evidence/categorize/route.ts:66
- **Scenario**: Four specs derive the case id with the identical expression `typeof record.caseId === "string" && record.caseId.trim() !== "" ? record.caseId.trim() : null`. Grep `typeof record.caseId === "string"` shows those four PLUS `categorize/route.ts:66`, which uses the SUBTLY DIFFERENT `typeof record.caseId === "string" ? record.caseId : null` — no trim, no empty-string-to-null collapse. So a `" "` or `"abc "` caseId is normalized away on draft/rfe/critique/forecast but passed through raw on categorize.
- **Root cause**: Copy-paste of the extraction snippet, with categorize predating (or diverging from) the trimmed form when it was migrated to the orchestrator.
- **Impact**: Mostly cosmetic duplication, but the categorize divergence means a whitespace-padded `caseId` reaches `evidence.getDocuments`/`addDocument` un-normalized, where the four other routes would have treated it as "no case." Low because the adapter's resolveCase will reject a non-matching id anyway, but it is an inconsistency a single helper would erase.
- **Fix sketch**: Add `parseCaseId(record: Record<string, unknown>): string | null` (alongside the existing `str`/`parseFocus` helpers in `criteria-text.ts`, or in `access.ts`) using the trimmed form, and call it from all five specs — folding categorize onto the same normalization.

## 5. `build`'s `… as unknown as Record<string, unknown>` double-cast repeated across every spec
- **Severity**: Low
- **Category**: cleanup
- **File**: src/lib/ai/operation.ts:129 (the `build` signature) — duplicated at draftOperation.ts:234-241, critiqueOperation.ts:128-132, forecastOperation.ts:155-159, rfe/route.ts:120-124, guidance:71-75, qualify:62-69, categorize:95-99 (8 occurrences, grep `as unknown as Record<string, unknown>`)
- **Scenario**: `build` is typed `(output, source, input) => Record<string, unknown>`, but every spec's domain builder (`buildDraftResult`, `buildRfeResult`, `buildQualifyResult`, …) returns a concrete typed result object whose shape is NOT assignable to `Record<string, unknown>` index-style, so each call site launders it through `… as unknown as Record<string, unknown>` (and several also cast `source as Parameters<typeof buildX>[1]`). The double-cast appears 8 times.
- **Root cause**: `build`'s return type was fixed as the open `Record<string, unknown>` (so the orchestrator can spread `...finalBody`), forcing every concrete builder result to be force-cast at the boundary. The result types are well-defined objects, not loose records.
- **Impact**: Purely cleanup, but `as unknown as` is the strongest possible "trust me" escape hatch — repeated 8× it suppresses any structural check that a builder actually returns an object-spreadable value, and it is noise every new spec must copy. A genuinely wrong builder return (e.g. an array or primitive) would type-check.
- **Fix sketch**: Make `build`'s declared return `Readonly<Record<string, unknown>>` OR (cleaner) widen the spec's `build` type to `(…) => Record<string, unknown> | TResult` constrained to object types, and add a tiny `asBody<T extends object>(v: T): Record<string, unknown>` single-cast helper in `operation.ts` so each spec writes `build: (o, s) => asBody(buildDraftResult(o, s))` — one audited cast instead of eight inline `as unknown as`.
